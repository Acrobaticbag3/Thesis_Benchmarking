These are fantastic results! The numbers perfectly illustrate the architectural differences between each abstraction strategy. This will make for great analysis in your thesis.

Here is exactly what these results mean when comparing the tools side-by-side:

1. kustomize_tool (The Lightweight Speedster)
Time (~1.4s deploy): Kustomize is the fastest to deploy. Because it is baked directly into kubectl, there is no secondary state management (unlike Helm's release secrets). It simply computes the YAML and blasts it to the API.
Overhead (79 MB / 39% CPU): It has a high CPU percentage because its total lifespan is so short. It spends a massive burst of CPU cycles computing the deep-merge overlays of your YAMLs, and then immediately exits. It uses 79 MB of RAM to hold those merged structures in memory before sending them off.
2. helm_tool (The Balanced Standard)
Time (~2.1s deploy): Slightly slower than Kustomize because Helm has to manage state. When it deploys, it must compile the Go templates, bundle them into a Secret, upload that Release state to the cluster, and then apply the objects.
Overhead (60 MB / 11% CPU): Helm is very optimized. It holds a 60 MB memory footprint to parse the templates. Because it runs slightly longer than Kustomize (due to state management), its CPU utilization averages out lower (11%).
3. timoni_tool (The Strict & Efficient Engine)
Overhead (53 MB / 3% CPU): Timoni is incredibly efficient! It requires the lowest amount of RAM (53 MB) and almost barely touches the CPU (3%). This proves that the CUE language's graph-unification engine is wildly optimized compared to standard text templating.
Time (~5.1s deploy): While rendering is instant, its elapsed time is 5 seconds. This is because Timoni uses Kubernetes Server-Side Apply and has a much stricter, more robust polling mechanism for its --wait command. Most of its 5-second lifespan is spent "sleeping" (0% CPU) while carefully verifying the cluster state.
4. cdk8s_tool (The Heavyweight Behemoth)
Time (~17.8s deploy / 22s rollback): CDK8s is dramatically slower.
Overhead (263 MB / 35% CPU): As we predicted! To deploy using cdk8s, your machine literally has to boot up a V8 Node.js engine, load the TypeScript compiler, synthesize the code into JSON, write it to disk, and then fire up a separate kubectl process to apply it.
The Insight: It takes ~5x more RAM than Timoni and maintains a 35% CPU utilization over a massive 17-second window. This means the Node.js process spent roughly 6 whole seconds maxing out a CPU core just to generate the manifests!
Thesis takeaway: If a company wants to use a general-purpose language like TypeScript (cdk8s) to write their Kubernetes configurations, they gain developer familiarity but pay a massive "computational tax" in their CI/CD pipelines compared to purpose-built, domain-specific languages like CUE (Timoni).
